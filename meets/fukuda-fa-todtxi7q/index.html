1. マネーフォワード（MF）の現状共有

  • 現状はデータ引き取りのみ。登録機能はなく効率化が進んでいない
  • MF MCPは出たが「使い物にならない」（福田さんの見解）
  • 6月にAPI公開予定。freeeと同等の自動登録機能が出てくる見込み
  • 7月にMF独自AIリリース予定。どこまで使えるかが今後の評価ポイント
  • freeeはリーダーアカウントで傘下事業所を一括管理できるが、MFは事業所ごとに個別APIキー付与が必要
  • 福田さんは現状2社のみAPI連携。残り8社はオーナーへのオンボーディングが難航中
  • freeeエージェントもMCPを廃止しない方針。会計データの集計・レポーティング用途でMCPは残る（直前のfreee担当者との打ち合わせ情報）

2. 一人運営60社の実態

  • スタッフなし、業務委託なし。完全に一人で運営
  • 面談は定例でなく「アジェンダがある時のみ」。税金・資金繰り・相談ごとが生じたタイミングで30分程度
  • 月次仮締め報告は行っていない。顧客から求められていないため
  • 「1社契約しても固定時間が増えない設計」が一人運営の核心
  • 顧客層：東京渋谷のIT系スタートアップ、25歳前後、VC調達直後（SEED期）が中心。赤字が多く消費税も1〜2期目でないケースがほとんど
  • 個人顧客は10人程度。残りは全員法人

3. AI自動仕訳の運用実態

  • 自動仕訳の目標値：7〜8割。残り3割は手動介入が必要
  • 手動が残る主なケース：給与・源泉仕訳、複合仕訳、コンビニ等の課税区分混在（課税10%・8%・非課税が同一取引先で混在）
  • クレカ明細は課税区分が判断できないので割り切り運用（コンビニは課税でまとめる等）
  • 資料が揃わない顧客が多いため、1万円以下の少額は簡易特例で割り切り。インボイス番号も割り切り
  • 「仕入れ税額控除をいくら取れなかったか報告してない。言ったところで向こうも出せないから」
  • 将来構想：領収書が揃った場合に仕訳帳と照合してズレをAIでピックアップする仕組みを検討中（未実装）

4. AI連携のセキュリティ・リスク整理

  • 畠山先生の現状連携：Google関係・Slack・Notionは接続済み
  • 福田さんの現状連携：カレンダーのみ。メール・Slack・Google Driveは未接続（セキュリティが心配）
  • プロンプトインジェクションの存在は認識済み。メール経由で見えない文字に攻撃コマンドを埋め込み、AIが実行してしまう攻撃
  • 「接続しているだけ」では流出リスクは限定的。AIがコンテンツを処理すると外部サーバーに渡るため、ハッキング時は流出リスクあり
  • 具体的な対策方針は未整理。セキュリティ専門家（勉強会の知り合い）に確認予定
  • 「情報が最も詰まっているメールだけは連携しない」という切り分けも一つの判断軸として言及

5. 集客戦略

  • X：毎日・タイムリーに届けられるメディア。税理士のX発信者が少ないニッチ。3000フォロワーから手応えを感じた
  • note：GoogleのSEOが強い。Google検索で自分の記事が上位表示。AIとの相性も良く、信頼資産として蓄積できる
  • Podcast：ファン獲得・言語化の訓練に有効。MCと編集者を月10〜13万で1年間外注して継続。現在約400フォロワー（コミット強いファン層）
  • 発信の位置づけ：言語化とリード獲得の両方
  • ターゲット：東京渋谷・IT系・VC調達・社会人経験なし25歳層
  • 継続の仕組み：自己流のX以外は「外部スタッフを使う・お金を払う」ことで強制力を作った（Podcast MC・note顧問 月5万円・約1年半）
  • 転換点：2022年NFTブームに乗って発信→フォロワー3000人→IT企業家層とのつながりが生まれた
  • 独立前から顧客開拓を意識。前職入社時に「自分の顧客を持ってくること」を条件として交渉していた

6. Taxsys・Comet（後日フォロー予定だったが会話で触れた）

2-3 新規取引・例外パターンの検知と取り込みループ

現状、明示的なループ設計は持っていないが、以下のアプローチで補完している：

  • 月次処理の中で未分類・要確認明細をリストアップし、目視確認後にfreeeのルール登録に反映
  • freeeの自動学習機能で次回以降に自動適用される

さらに自動化するなら：

  • 摘要キーワードと科目・税区分のルール表をgit管理し、新規パターンが出た際に「Claude提案→承認→ルール表更新」フローを組む
  • Claude Haiku APIに1件ずつ投げて判定根拠と共に候補を返させ、最終確認は畠山先生が行う設計

3-2（補足） プロンプトインジェクションへの具体的対策

  1. メールは接続しない：外部からの不審なコンテンツが直接AIに渡る最も危険な経路。Google Drive・Slack等より優先的に除外を検討
  2. MCPの権限はread-onlyを基本に：AIが自律的に書き込み・送信できる権限は最小化する。write操作は人間の承認を必須にする
  3. 外部コンテンツを渡す際は最小化：「このメールの件名と差出人だけ教えて」のように情報を絞る。本文を丸ごと渡さない
  4. 連携先の定期棚卸し：使っていないMCP接続は定期的に切断する

3-3 税理士法38条の射程と外部AIへの渡せる/渡せない判断軸

税理士法38条は「業務上知り得た秘密の漏示禁止」。外部AIサーバーへの送信は形式上「漏示」に該当しうるが、以下の整理で実務対応が可能。

Anthropicのリスク評価：APIの入出力データはモデル学習に使用しないことを利用規約に明記。通信はTLS暗号化、SOC2 Type II取得済み。

弊所が採用している判断軸：

渡してよいもの渡さないもの
摘要キーワード（パターン分類用） 具体的な取引先名・金額・口座番号
「この種類の取引は何費か」という判断ロジックの問い合わせ 生データの仕訳帳・試算表全体
一般的な税務知識の確認 顧客名が特定できる情報

契約書での手当て：顧問契約書に以下のAI活用条項を追記することを推奨。

「業務遂行にあたりAIツールを使用する場合がある。外部AIサービスへのデータ送信は必要最小限とし、利用するサービスはAnthropicの利用規約に基づきデータが学習利用されない体制を採用している」

弊所の現行条項を後日チャットで共有する。

後3 顧問契約書のAI活用条項（弊所契約書 第16条の2 全文）

弊所の顧問契約書（税務会計業務委託契約書）に組み込んでいるAI活用条項の全文は以下のとおり。福田さんの現行契約書との比較・追記の参考にどうぞ。

第16条の2（生成AIツールの利用）

依頼人は、税理士が本税務会計業務の遂行にあたり、生成AI（人工知能）ツール（Anthropic社のClaude、Google社のGemini等のサービスを含むがこれらに限られない。以下「生成AIツール」という。）を利用することについて、予め理解し、承諾する。

2. 税理士は、生成AIツールを次の各号に定める業務に利用することができる。

  1. 記帳代行業務（仕訳の分類、入力及び確認を含む。）
  2. 税務及び会計に関する調査並びに情報収集
  3. 依頼人からの質問又は相談に対する回答の作成及び補助
  4. その他本税務会計業務の遂行に合理的に必要と認められる業務

3. 依頼人は、前項の生成AIツールの利用に伴い、依頼人に関する情報（取引データ、財務情報、仕訳データ、取引先名等を含む。）が、生成AIツールを提供する事業者（以下「AIサービス提供者」という。）のサーバー（日本国外に所在するものを含む。）に送信される場合があることを予め理解し、承諾する。

4. 税理士は、生成AIツールの利用にあたり、次の各号に定める事項を遵守するものとする。

  1. AIサービス提供者との間で、依頼人の情報がAIの学習データとして使用されない設定又は契約条件を確保するよう努めること
  2. 生成AIツールへの情報の入力を、本税務会計業務の遂行に合理的に必要な範囲に限定すること
  3. 生成AIツールが出力した結果については、税理士の責任において正確性及び妥当性を確認した上で使用すること
  4. 前条（秘密情報の取扱い）に定める義務は、生成AIツールの利用においても適用されること

5. 生成AIツールの利用に起因し又は関連して依頼人に損害が生じた場合における税理士の責任は、第14条（損害賠償）の規定に従うものとする。

ポイント整理（3条の対応）
① 「日本国外サーバーへの送信」について依頼人の事前同意を取得（第3項）
② 「学習データとして使用されない設定を確保するよう努める」義務を税理士側に課す（第4項1号）
③ 秘密情報の守秘義務（第16条）はAI利用においても適用される旨を明示（第4項4号）
④ AI起因の損害賠償は第14条（上限：支払済み報酬の合計額）に準拠

後4 業務ログ・作業ログ収集の仕組み
